# **Реализация отказоустойчивой распределённой системы на базе Kubernetes**

## **Цель работы**
Разработать отказоустойчивую распределённую систему для взлома MD5-хэшей методом перебора (brute-force), обеспечивающую надёжную обработку запросов пользователя даже при частичных сбоях компонентов системы. Система должна быть развернута в окружении Kubernetes и использовать современные подходы к отказоустойчивости, масштабируемости и сохранности данных.

---

## **Общее описание системы**

Система предназначена для поиска строки, соответствующей заданному MD5-хэшу, методом перебора всех возможных строк заданной максимальной длины, составленных из определённого алфавита.

Система состоит из следующих компонентов:
- **Менеджер (Manager)** — центральный сервис, принимающий запросы от пользователей, управляющий задачами и координирующий работу воркеров.
- **Воркеры (Workers)** — вычислительные узлы, выполняющие перебор строк и вычисление их хэшей.
- **RabbitMQ** — брокер сообщений для асинхронного взаимодействия между менеджером и воркерами.
- **MongoDB** — отказоустойчивое хранилище данных для сохранения состояния запросов и задач.

Все компоненты разворачиваются в кластере **Kubernetes**.

## **Функциональные требования**

### **1. Пользовательское API (REST, JSON)**

Менеджер предоставляет REST API для взаимодействия с клиентом.

#### **Запрос на взлом хэша**
```
POST /api/hash/crack
```
**Тело запроса:**
```json
{
  "hash": "e2fc714c4727ee9395f324cd2e7f331f",
  "maxLength": 4
}
```
**Ответ (успешный):**
```json
{
  "requestId": "730a04e6-4de9-41f9-9d5b-53b88b17afac"
}
```

#### **Проверка статуса запроса**
```
GET /api/hash/status?requestId=730a04e6-4de9-41f9-9d5b-53b88b17afac
```

**Ответ (обработка в процессе):**
```json
{
  "status": "IN_PROGRESS"
}
```

**Ответ (строка найдена):**
```json
{
  "status": "READY",
  "results": ["abcd"]
}
```

**Ответ (строка не найдена):**
```json
{
  "status": "READY",
  "data": []
}
```

**Ответ (ошибка/таймаут):**
```json
{
  "status": "ERROR"
}
```

---

### **2. Алфавит и генерация строк**

- Используется алфавит: строчные латинские буквы `a-z` и цифры `0-9` (всего 36 символов).
- Строки генерируются длиной от 1 до `maxLength` включительно.
- Воркер не должен хранить все комбинации в памяти — необходимо использовать итеративный подход (например, генерация по одной комбинации за раз).
- Менеджер разбивает пространство задач на части фиксированного размера (например, до 100 000 комбинаций на задачу).
  - Пример: если общее количество комбинаций — 1 500 000, то будет создано 15 задач по 100 000 комбинаций каждая.
- Каждая задача содержит:
  - Начальный индекс в последовательности комбинаций.
  - Количество комбинаций для обработки.
  - Целевой хэш.
  - Максимальную длину строки.

---

### **3. Взаимодействие через RabbitMQ (JSON)**

Все взаимодействия между менеджером и воркерами происходят асинхронно через **RabbitMQ** с использованием **JSON** в качестве формата сообщений.

#### **Очереди и обменники**
- `task.queue` — для отправки задач от менеджера к воркерам.
- `result.queue` — для возврата результатов от воркеров к менеджеру.
- Обе очереди используют **Direct Exchange**.
- Все сообщения должны быть персистентными (сохраняются при перезапуске RabbitMQ).
- Используется механизм acknowledgement: сообщение удаляется из очереди только после успешной обработки воркером.

---

#### **Формат задачи (от менеджера к воркеру)**
```json
{
  "taskId": "70c26e4e-0d51-11f1-a84d-73f4fbe1eca2",
  "requestId": "730a04e6-4de9-41f9-9d5b-53b88b17afac",
  "startIndex": 0,
  "count": 100000,
  "targetHash": "e2fc714c4727ee9395f324cd2e7f331f",
  "maxLength": 4
}
```

#### **Формат результата (от воркера к менеджеру)**
```json
{
  "taskId": "70c26e4e-0d51-11f1-a84d-73f4fbe1eca2",
  "requestId": "730a04e6-4de9-41f9-9d5b-53b88b17afac",
  "results": ["abcd"],
  "status": "DONE"
}
```
Возможные значения `status`: `"DONE"`, `"ERROR"`.

## **Требования к отказоустойчивости**

### **1. Отказоустойчивость менеджера**
- Все данные о запросах и задачах должны храниться в MongoDB.
- Менеджер не должен отвечать клиенту, что задача принята, пока она не будет сохранена в базу, и репликация подтверждена.
- При перезапуске менеджера он должен:
  - Восстановить все незавершённые запросы из MongoDB.
  - Перепроверить состояние задач и при необходимости повторно отправить невыполненные задачи в очередь.
  - Обработать накопленные результаты из `result.queue`.

---

### **2. Отказоустойчивость MongoDB**
- MongoDB должна быть развернута как реплика-сет в Kubernetes.
- Конфигурация: одна primary, две secondary ноды.
- При отказе primary ноды должен происходить перевыбор primary.
- Все операции записи должны подтверждаться минимум на двух нодах (write concern: majority).

---

### **3. Отказоустойчивость воркеров**
- В системе должно быть не менее двух воркеров, развернутых как отдельные поды в Kubernetes.
- Если воркер падает во время обработки задачи, сообщение автоматически возвращается в очередь (благодаря `ack`), и другой воркер берёт его на обработку.
- Допускается повторная обработка одной задачи (например, если воркер обработал, отправил результат в очередь, но не смог отправить ack). Менеджер должен корректно обрабатывать такие случаи.

---

### **4. Отказоустойчивость очередей**
- Все сообщения в RabbitMQ должны быть персистентными.
- При перезапуске RabbitMQ сообщения не должны теряться.
- Если менеджер не может отправить задачу в очередь (например, RabbitMQ недоступен), он сохраняет задачу в MongoDB со статусом `QUEUED` и повторяет отправку при восстановлении соединения.
- При старте менеджер должен проверить наличие непереданных задач в базе и отправить их в очередь.

---

## **Технические требования**

- Все компоненты (менеджер, воркеры, RabbitMQ, MongoDB) должны быть развернуты в едином кластере Kubernetes.
- Настроить мониторинг:
  - RabbitMQ: статус, количество сообщений в очередях;
  - MongoDB: статус/роли реплик;
  - Воркеры: статус (выполняет задачу / простаивает), рейт успешных и неуспешных задач;
  - Менеджер: статус, рейт запросов от клиентов в разбивке по ручкам (crack, status) с кодами ответов.
- Количество реплик воркеров должно быть настраиваемым.
  - Опционально: сделать автоматическое масштабирование количества воркеров в зависимости от количества задач в очереди.
- Менеджер и воркеры могут быть выполнены на любом современном языке программирования.

## **Ожидаемое поведение при сбоях (тестовые кейсы)**

| Сценарий | Ожидаемое поведение |
|--------|---------------------|
| Остановка менеджера | После перезапуска менеджер восстанавливает все запросы и задачи из MongoDB, обрабатывает накопленные результаты. |
| Остановка primary ноды MongoDB | Происходит автоматический failover, система продолжает работу. |
| Остановка RabbitMQ | После перезапуска все персистентные сообщения остаются в очередях, обработка продолжается. |
| Остановка одного воркера во время обработки | Задача возвращается в очередь и выполняется другим воркером. |
| Остановка всех воркеров | Задачи остаются в очереди, ожидают появления воркеров. |
| Отказ сети между менеджером и RabbitMQ | Менеджер сохраняет задачи в MongoDB и отправляет их при восстановлении. |
